# Homolog Server Deploy

Pricely homolog server deploy runs the full Docker stack:

- PostgreSQL
- Redis
- pgAdmin
- backend
- web

The workflow is manual: `.github/workflows/deploy-homolog-server.yml`.

## Environment Split

The deploy intentionally separates three env surfaces:

- Compose infrastructure env: generated by the workflow as `.env` on the
  server. It contains image names, tags, ports, Postgres credentials, pgAdmin
  credentials, and public service URLs consumed by Docker Compose.
- Backend runtime env: stored in GitHub as `BACKEND_ENV_FILE` and written to
  `.env.backend` on the server. It contains backend-only application settings.
- Web runtime env: optionally stored in GitHub as `WEB_ENV_FILE` and written to
  `.env.web` on the server. It contains web-only `VITE_*` settings that are not
  already covered by repository variables.

Do not duplicate generated infrastructure values in `BACKEND_ENV_FILE`.
`DATABASE_URL`, `REDIS_HOST`, `REDIS_PORT`, `PORT`, `APP_HOST`, `NODE_ENV`, and
`WEB_APP_URL` are injected by `docker-compose.homolog.yml`.

## Required Repository Secrets

- `DEPLOY_HOST`
- `DEPLOY_USER`
- `DEPLOY_SSH_KEY`
- `GHCR_TOKEN` or `GHCR_PAT`
- `BACKEND_ENV_FILE`
- `POSTGRES_PASSWORD`
- `PGADMIN_DEFAULT_PASSWORD`

`RUNTIME_ENV_FILE` is still accepted by the workflow as a legacy fallback, but
new configuration should use `BACKEND_ENV_FILE`.

`BACKEND_ENV_FILE` is written to `.env.backend` on the server and should contain
backend-only runtime values that are not generated by Compose, such as:

```text
JWT_ACCESS_SECRET=...
LOG_LEVEL=info
QUEUE_WORKERS_ENABLED=true
PUBLIC_SEARCH_METRICS_WINDOW_SIZE=500
PUBLIC_SEARCH_P95_TARGET_MS=750
PUBLIC_SEARCH_PG_TRGM_MIN_SAMPLES=100
```

Do not put CPF, receipt personal data, or local developer-only secrets in this file.

Optional secrets:

- `WEB_ENV_FILE`
- `TS_OAUTH_CLIENT_ID`
- `TS_OAUTH_SECRET`

`WEB_ENV_FILE` is written to `.env.web` on the server. Keep it limited to web
runtime settings, for example:

```text
VITE_AUTH_BASE_PATH=/auth
VITE_REGIONS_BASE_PATH=/regions
```

`VITE_API_BASE_URL` should normally remain a repository variable because it is
public configuration and is also injected explicitly by the Compose env.

## Current Expected Repository Configuration

Minimum secrets:

```text
DEPLOY_HOST
DEPLOY_USER
DEPLOY_SSH_KEY
GHCR_PAT
BACKEND_ENV_FILE
POSTGRES_PASSWORD
PGADMIN_DEFAULT_PASSWORD
```

Minimum variables:

```text
DEPLOY_PATH=/srv/stacks/pricely
VITE_API_BASE_URL=https://<backend-public-url>
VITE_ALLOWED_HOSTS=pricely.grmeireles.dev
VITE_ALLOW_ALL_HOSTS=true
WEB_APP_URL=https://<web-public-url>
```

`VITE_ALLOW_ALL_HOSTS=true` is the homolog default because the web container
runs the Vite development server behind Cloudflare Tunnel. Set it to `false`
only if every public tunnel host is covered by `VITE_ALLOWED_HOSTS`.

Use `GHCR_TOKEN` instead of `GHCR_PAT` only if the token has package pull access
for the generated GHCR images.

## Database Behavior

The server compose never force-resets the database on normal container start.

Workflow inputs control database maintenance:

- `reset_database=false`: runs the safe versioned migration bootstrap. Existing
  databases without Prisma migration history are baselined only when their live schema
  exactly matches `schema.prisma`; drift aborts the deploy.
- `reset_database=true`: runs `prisma migrate reset --force --skip-seed`; this is
  destructive.
- `run_seeders=true`: runs `npm run db:seed` after schema sync/reset.

Use `reset_database=true` only when the homolog server data can be discarded.

The backend container does not mutate the schema on ordinary startup. Schema changes
are applied by the deploy workflow before the backend/web containers are recreated.
